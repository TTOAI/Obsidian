### 트렌드 대응 규칙
- 더 좋은 AI 모델, agent, framework 등이 나온 경우 어떻게 할 것인지
- 무시, 완전 재시작 등

### 그럼에도 필요한 것들
- 피드백 그룹: 개발 내용을 꾸준히 공유하고 피드백을 받을 수 있는 그룹
- 개발 커뮤니티: 트러블이 생겼을 때나 너무 막막할 때 대화할 수 있는 커뮤니티

### AI First 작업 구조
- 파이프라인: AI가 작업한 코드는 어떻게 서비스 되는가
- 모니터링 시스템: AI 코드 검증 방법

### AI First 스택과 인프라
- 오픈소스, 미리 만들어진 스택을 활용 (잘 만들어진)
- 비용보다 속도
- 미래보다 현재
- 완성보다 출시
- 모든 것은 가역적

### 프로그래밍 언어 선택
- Typescript: high하지만 strict하므로 상대적으로 loose한 javascript, python보다 에러에 대한 모니터링이 수월함
- Go, Rust

### 모니터링 단계 세팅
- 코드 설계 단계: 아키텍트 충실히 합의
- 코드 생성 단계: 생성 코드와 생각 과정을 빠르게 눈으로 따라감
- Linter 및 검증 프레임워크 세팅은 기본
- Rules 세팅
	- ![[Pasted image 20250602170438.png]]
- Hot Reload가 지원되면 좋음 (build cycle 짧게)
- 테스트 코드는 필수 (Human TDD vs. AI Scenario)
	- 시나리오 기반 테스트 코드 작성
- CI/CD 레벨 검증 시나리오 필수
- Automated Test 도구들 (fastlane, firebase test lab 등)
- 디버깅 리포트 도구 (Sentry)
**-> 결론: 검증은 많을수록 좋다**

## DDD와 철저한 Atomic 모듈화
- 모듈: hierarchy 존재, 스펙 존재, interface 존재
- Context Size가 커지면 -> 일의 마디와 계획 잡기 힘듦, 비용 증가, 에러 확률 올라감
- DDD App 단위는 최소한의 Atomic한 기능만
- 모듈 최소, 대신 Interface 패턴 확보 (모듈끼리의 소통 패턴, 방식)

## 네이밍 컨벤션과 주석에 주의

	- 네이밍 시 명확한 관계와 거리 설정
