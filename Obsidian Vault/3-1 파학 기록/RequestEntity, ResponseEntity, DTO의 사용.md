
## RequestEntity<>, ResponseEntity<> 사용 이유
- 세밀한 HTTP 제어가 필요할 때
	- HTTP 요청 및 응답의 헤더, 바디, 상태 코드, URI 등 메타데이터를 직접 다루어야 할 경우
- 커스텀 헤더 및 상태 코드 설정
	- 클라이언트로부터 특정 헤더 정보를 받아 검증할 때
	- 응답 시 커스텀 헤더를 추가하고 싶은 경우
- 요청 전체 정보를 캡슐화
    - 요청 바디뿐만 아니라 헤더 등의 부가 정보를 함께 다룰 때

## DTO (Data Transfer Object) 사용 이유
- 계층 간 데이터 전송의 분리
	-     클라이언트와 서버 사이, 또는 서비스 계층과 데이터 계층 간의 데이터 전달 시 내부 도메인 모델(Entity)과 API에 노출하는 데이터를 분리할 수 있음
- 보안 및 유지보수
    내부 모델의 민감한 정보가 외부에 노출되지 않도록 하고, 필요한 필드만 전달하여 API 계약을 명확하게 함
- 데이터 검증 및 변환
    - 요청 데이터를 DTO에 매핑하여 검증하고, 비즈니스 로직에 맞게 엔티티로 변환할 수 있음
    - 응답 데이터를 DTO로 구성하여 클라이언트에 필요한 데이터만 노출함
